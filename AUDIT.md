# AUDIT.md — Job Application Tracker (read-only deep audit, 2026-07-19)

Scope: entire repo at commit `4f5d048` (branch `main`, clean tree). Every finding below cites file:line and was verified by reading the code — findings I could not confirm are in section D, not asserted. "This is broken" and "I'd do it differently" are kept separate: the findings table (B) contains only defensible defects; taste-level notes are marked as such or omitted.

**Orientation summary (Phase 1):** Free, donation-supported Kanban job-application tracker. React 19 + Vite + TS + Tailwind 4 + dnd-kit, Supabase (Postgres/Auth/Edge Functions), IndexedDB (`idb`) for local-first guest mode. MVP per `job-tracker-mvp-brief.md` is complete; post-MVP features shipped (multi-tracker, archive filters, account panel, AI screenshot extraction via Claude Haiku). Live at jobtracker.fazare.dev; auto-deploys from `main` via Cloudflare. `PLAN.md`/`HANDOFF.md` claim everything done and verified — the repo state matches that claim (clean tree, all pushed, `npm audit` clean, and the code even passes `tsc --strict` although strict is off). Where docs and code genuinely disagree is the **privacy policy** (see H2) and the **export scope** (see M1).

---

## A. Executive summary

1. **A real guest-data-loss path exists** despite "never lose their data" being the product's #1 principle: sign out → use as guest → log back in → guest data is silently never uploaded (stale `migrated` flag) and is then permanently wiped by the next sign-out's `clearLocalStore()`. (H1)
2. **The live privacy policy is factually wrong twice**: sign-up now requires a name, and AI extraction sends user screenshots to Anthropic — while the page claims only email+password are collected and data is never shared with third parties. Live EU-facing product → compliance exposure. (H2)
3. **Failed writes are silent almost everywhere**: drag/drop, create, archive, undo, delete have no error surfacing; the drag path even applies an optimistic update that is never reverted, so the UI shows a saved state that wasn't saved. (H3)
4. **There is no password-reset flow at all.** A live email+password product where a forgotten password means permanent lockout (support channel: none). (C1 — biggest missing piece)
5. **Zero automated tests** — including for the migration code the brief itself calls the highest-risk, correctness-critical part. All verification is manual QA that will not be re-run after future changes. (H4)
6. **Data export omits trackers** — the GDPR export and "guest safety net" cannot reconstruct the user's boards (`tracker_id` references nothing in the file), and there is no import to restore it anyway. (M1)
7. **The extraction spend-cap has a check-then-act race**: parallel requests all pass the quota check before any event row is inserted, so the per-user and global caps — the stated wallet protection — are bypassable. (M2)
8. Board card order and tracker tab order are **nondeterministic** (no `ORDER BY` on any Supabase read, no client-side sort on the board) — cards/tabs can shuffle between refreshes. (M3)
9. **Ctrl/Cmd+Z is globally hijacked** for 10 s after any archive — including while typing in a text field, where it unarchives instead of undoing typing. (M4)
10. Good news, verified: RLS policies are correct and complete for the app's write patterns (one narrow exception, L2), no secrets in git history (only the intentionally-public publishable key), `npm audit` reports zero vulnerabilities, Edge Functions verify passwords server-side, and the delete-account RPC is safely self-scoped.

## C. Missing pieces (for an app of this type and stage)

1. **Password reset.** Verified absent (`grep resetPassword|forgot` → nothing; `AuthModal.tsx` has only sign-up/log-in). A live email+password product with no "Forgot password?" means a forgotten password is permanent lockout — and there's no support contact anywhere either. Supabase's `resetPasswordForEmail` + a redirect handler is a small, contained feature. **This is the single most important missing piece.**
2. ~~**Import / restore.**~~ **Ruled out, 2026-07-21 — not a gap, a deliberate exclusion.** Export exists as "the guest safety net" but nothing can read the file back, which does leave a guest whose browser is cleared with an unusable JSON file. But the fix creates a worse problem: `PER_USER_MONTHLY_LIMIT` is scoped to `user_id`, not to the person, so import would let anyone export their board, make a fresh account, and re-import it for another 20 free extractions — repeatably. Don't build this without solving that reset vector first. See `PLAN.md`, "Out of scope for MVP."
3. ~~**Automated tests + CI.**~~ **Done.** `.github/workflows/ci.yml` runs `tsc && oxlint && vitest` on every push.
4. ~~**Error telemetry.**~~ **Done, scoped down, 2026-07-21.** No error boundary (M8) is now fixed (`src/components/ErrorBoundary.tsx` exists). This item — actual logging/reporting of client-side failures — is now also addressed, but deliberately local-only: `src/lib/globalErrors.ts` catches uncaught errors/rejections and routes a generic notice through the existing toast; console.error carries full detail. No third-party SDK, no remote sink — the user explicitly rejected that scope because a telemetry processor would be a new data recipient to disclose in the privacy policy this same session just fixed (H2).
5. ~~**Any support/contact channel.**~~ **Done, 2026-07-21.** `fazare@fazare.dev` (Zoho) is now in the privacy page, the app footer, and the account-deletion email.
6. Minor: quota indicator for extraction (L8) — **done**, `ApplicationForm.tsx` shows "N of 20 free extractions left". An in-form hint that extraction is signed-in-only — **deliberately deferred**, per the user: extraction is one of the product's most important features and deserves a properly designed hint, not a one-line afterthought. `stage_history` is written everywhere but read nowhere — unchanged, still fine (brief defers analytics).

Not missing (checked, in good shape): RLS coverage (all four ops on `applications`/`trackers`, scoped subqueries on `stage_history`), secrets hygiene (git history contains only the public publishable key; `.env` never committed), dependency health (`npm audit`: 0 vulnerabilities), spend-capped AI with server-side auth, self-scoped `SECURITY DEFINER` deletion, GDPR delete + export skeleton, guest mode with no signup wall, keyboard path for drag (dnd-kit), donation link, privacy page existing at all.

## D. Needs verification (I could not confirm from the repo)

1. ~~**Supabase spend cap actually ON**~~ — **moot, resolved 2026-07-21.** The project is on Supabase's free plan, which has no usage-based billing to cap in the first place (hard resource limits/throttling instead, not a bill). The brief's §3 wallet-protection concern is satisfied by "no paid plan, no card on file," not by a spend-cap toggle — revisit if the project ever upgrades to a paid plan, since a spend cap would then be a real, separate thing to configure.
2. **`sb_secret_` key rotation** — PLAN.md ("Decisions & notes") says it was shared once in chat and "should probably be rotated." No evidence either way. If not rotated, do it.
3. ~~**Deployed Edge Function code = repo code**~~ — **resolved, 2026-07-21, live-confirmed.** Both functions now carry a `FUNCTION_VERSION` constant on every response (including unauthenticated `OPTIONS`). User redeployed both, diffed the dashboard's prior code against the repo first (no drift found at that point), then confirmed via `curl -sI -X OPTIONS <url>` that `x-function-version` matches the repo constant on both. Caveat stands for the future: only detects drift if the constant is bumped on every deploy.
4. ~~**Sign-up with an already-registered email**~~ — **fixed and live-confirmed, 2026-07-21.** `AuthModal.tsx`'s post-signup panel no longer asserts a confirmation link was sent (which was false whenever the email was already registered — Supabase's anti-enumeration design returns success either way). The governing dashboard setting is **"Rate limit for sign-ups and sign-ins": 30 requests/5 min per IP (360/hour)** — distinct from the adjacent "token refreshes" and "token verifications" (OTP/magic-link) rows, which don't apply to `signInWithPassword`. Since the check runs inside the Edge Function, that "per IP" budget is almost certainly the function's own shared egress IP across every user, not scoped per attacker — confirms rather than resolves the false-lockout tension already noted here.

~~**New, found while investigating D4, not requested by this section**~~ — **fixed same day, 2026-07-21.** `signUp()` set a `job-tracker:pending-signup` flag unconditionally, including on Supabase's anti-enumeration success-for-an-existing-email response, where no account and no session ever arrived. The flag was only consumed when local guest data existed at the next session-start, so in every other case it lingered in `localStorage` indefinitely — and the next unrelated log-in on that browser that *did* have guest data auto-migrated it with no prompt. This reproduced M6 through a path the original M6 fix didn't cover. Fixed by validating the flag's `{email, at}` against the arriving session and always consuming it on session arrival, not only when guest data exists. See `PLAN.md`'s "Decisions & notes" for full detail.

## E. Recommended order of work

1. **H2 (privacy policy text)** — live compliance exposure, zero risk, ~30 minutes. No reason it should wait on anything.
2. **C1 (password reset)** — highest user-harm-per-week of the gaps in a live product; independent of everything else.
3. **H1 + M6 together (migration trigger semantics)** — same root cause (migration keyed on session-start + destructive `clearLocalStore`), so design once: decide when migration *should* run, then fix flag/clearing behavior. Do before adding any more data features; every future feature inherits this flow.
4. **H3 + M8 (error surfacing + error boundary)** — one shared toast/boundary mechanism, then thread the existing throw-paths through it. Do before H4 so tests can assert the failure behavior you actually want.
5. **H4 (test harness + the four high-leverage test targets)** — after 3 and 4 so the most dangerous logic is stable enough to pin. Add the `tsc`+lint CI gate here too (C3) — cheap while you're in tooling.
6. **M1 (trackers in export), M2 (atomic quota), M5 (revoke sessions on password change)** — independent, small-to-medium, in that order: M1 is a data-completeness promise already broken, M2 is a real cost hole but needs a hostile actor, M5 needs the GoTrue admin-logout verification first (D5-adjacent).
7. **M3, M4, M7, and the Lows** — quality pass; none block the above. L6 (`strict: true`) is a one-line change you can slip into any of these.

Sequencing logic: legal exposure first; then the thing locking real users out; then root-cause data-flow fixes before the test net that would otherwise pin down wrong behavior; then everything else, cheapest-per-risk first.
