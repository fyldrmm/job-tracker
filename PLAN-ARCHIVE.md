#### Bugs found and fixed after the checklist was first run

1. **Sign-out didn't clear the local cache** — fixed in `Board.tsx`'s `handleSignOut` (clears local store, resets active tracker, then signs out). User confirmed fixed (re-ran the checklist, sign-out step passed).
2. **Sign-up showed a bare "{}" error** — root cause confirmed via Supabase Auth logs: Resend's sandbox mode only delivers to the account owner's own verified email until a domain is verified at resend.com/domains; the signup attempt used a different real address (`fyildirimo2012@gmail.com`, the user's own secondary email — not a typo, corrected after initially misreading it as one) than the primary account owner's, which Resend rejected with a 550. Not a code bug — the AuthModal error-display hardening from the earlier commit stands regardless (never shows a bare "{}" again). **Resolved for real**: user bought `fazare.dev`, verified it on Resend, and updated Supabase's SMTP sender — sign-up with any real email now works, not just the account owner's own address.
3. **Data looked lost until a manual page reload after signup/login** — found during the SAME checklist re-run that confirmed #1 and #2. Root cause: the guest→account migration effect only refreshed `applications` after completing, never `trackers`. `useTrackers` has its own fetch triggered independently by the same `userId` change that starts migration, which could resolve on stale/empty data before migration's uploads landed, and nothing forced a second look once migration actually finished. Fixed by refreshing both applications and trackers once migration resolves. **User-confirmed fixed** (re-tested sign-out → sign back in, data present without a manual reload).
4. **Duplicate "My Applications" trackers piling up** — user noticed multiple unexpected tabs (including two identically-named "My Applications" ones) after repeated guest testing + eventual sign-up. Root cause: `useTrackers.refresh()` auto-created a "My Applications" tracker any time the list came back empty, including a brand-new guest's very first load with zero data — every fresh guest session (new browser/incognito/cleared storage) silently spawned its own tracker with a random UUID, and since they never merge by name, they all piled into the account once migrated. Fixed by removing the auto-create-on-empty behavior entirely: an empty tracker list is now a legitimate, expected state, surfaced as an explicit "Create your first tracker" button in `Board.tsx` (gated the header's "+ Add application" and the old empty-state CTA on `activeTrackerId` being set, since neither makes sense with zero trackers). Verified in-browser: fresh guest state shows the create-first-tracker prompt (no phantom tab), clicking it creates exactly one tracker, and a reload doesn't spawn a second one. Any pre-existing duplicate trackers from before this fix (e.g. the user's own test account) are just leftover data — safe to delete manually via the UI's tracker delete ("x" on each tab) whenever convenient, not a data-integrity issue.

#### A first attempt broke and was fully reverted — here's what happened and what changed the second time

The first build wired `change-password` straight into the UI without testing it against the real deployed function first. It crashed in the app with a generic "Edge Function returned a non-2xx status code" error for both right and wrong passwords, and diagnosing it took a long back-and-forth (terminal quoting issues with `curl`, shell history-expansion breaking on a password containing `!`, wrong assumptions about which token to use) before finding the real root cause was still unconfirmed. Rather than keep patching blind, the whole M7 attempt was reverted with `git checkout ba09e92` (the last fully-verified commit) and rebuilt from scratch, this time with a hard rule: **every Edge Function action gets curl-tested standalone, with a real account, before any UI touches it.**

That discipline caught the actual bug fast the second time: `supabase.auth.updateUser()` and `supabase.auth.signOut()` both throw `"Auth session missing!"` when called on a client built from just an `Authorization` header override (`global.headers`) — they need an actual session set via `setSession()`, which requires a refresh token the Edge Function never has (only the access token, via the header). Confirmed by hitting the exact error live, not guessed. Fixed by:
- Using the **service-role admin API** (`admin.auth.admin.updateUserById`) for the password change itself, since the caller's identity was already independently verified via `getUser()` and the password check before this point.
- Using a **raw `fetch()` to `/auth/v1/logout?scope=others`** (the same REST endpoint the JS SDK calls internally) instead of `supabase.auth.signOut()`, since that only needs the bearer token in the header, not the SDK's internal session state.

Sources checked before implementing (not guessed): [Supabase docs — Signing out](https://supabase.com/docs/guides/auth/signout), [GitHub discussion on the "Auth session missing" error and `admin.updateUserById`](https://github.com/orgs/supabase/discussions/26791).

Both the `delete` and `change-password` cases were confirmed working via direct `curl` calls against the real deployed function (wrong-password rejection for both, and a real successful password change with `{"success":true}`) before `remoteStore.ts`/the UI were wired to it.

#### A second bug found during live QA itself: password-change silently broke deletion

After the above was deployed and curl-verified, the live QA pass (deleting the primary account and signing up fresh) hit a real, reproducible bug: after successfully changing the account's password through the app, trying to delete that same account failed with `{"error":"Not authenticated"}`.

Diagnosed live, not guessed: called Supabase's own `/auth/v1/user` endpoint directly with the exact token the browser was using, which returned `"error_code":"session_not_found","msg":"Session from session_id claim in JWT does not exist"` — the token wasn't expired, but its session had been revoked server-side. Checked GoTrue's (Supabase Auth's) own source code to confirm why: the `scope=others` logout handler falls back to a **full global logout** (every session, including the current one) if it can't resolve "which session made this request" from context — and that's exactly what the M7 change-password feature's raw `fetch()` to `/auth/v1/logout?scope=others` had been hitting. The very act of changing your password was silently logging you out entirely, not just other devices, blocking whatever you tried to do next in that same session.

Fixed by removing the sign-out-other-devices feature entirely, rather than continuing to debug a mechanism that had just caused confirmed real damage to a live account. It was scoped-up beyond the original ask during planning; password-change itself doesn't depend on it. **User-facing impact if this had shipped further:** every user who changed their password would have been unexpectedly signed out immediately after, with no indication why.

Confirmed working end-to-end after the fix: fresh sign-up with a compulsory name, name showing in the sidebar, name editing, password change with wrong/right current password, and account deletion — all live, all by the user, on real accounts.

Root cause, confirmed by reading `@supabase/auth-js`'s actual installed source (`GoTrueAdminApi.signOut`) rather than guessing: `admin.auth.admin.signOut(jwt, scope)` is a thin wrapper over `POST /auth/v1/logout?scope=X`, and `scope=others` needs GoTrue to resolve "which session is the calling one" from the JWT so it can exclude it — that resolution is exactly what failed live. `scope=global` has no such resolution step at all; it's documented as unconditionally terminating every session for the user, which is also exactly what M5 wants (kill everything including the caller's own session, then force a clean re-login) — so this isn't a workaround for the old bug, it sidesteps the entire failure class by not depending on the thing that broke. `account-action/index.ts`'s `change-password` case now calls `admin.auth.admin.signOut()` with the caller's own already-validated access token (stripped of the `Bearer ` prefix) and `scope: 'global'`, after the password update succeeds; a revoke failure is logged but doesn't fail the response, since the client forces its own local sign-out unconditionally regardless of what the server did. **Confirmed caveat, from Supabase's own docs, not assumed:** this revokes refresh tokens (blocks future refresh and any future login with the old password) but does **not** retroactively invalidate an already-issued access token before its own `exp` claim — "the user won't be immediately logged out and will only be logged out when the Access Token expires." - **C4 note — global error handlers, and a real bug the tests found:** see `src/lib/globalErrors.ts` (`d2db730`) for the design (module-level pub/sub, not React context — zero contexts exist in this codebase and there's exactly one consumer). Worth a general lesson: `handleError` originally had `if (event.target && event.target !== window) return`, meant to guard against non-bubbling resource-load errors (`<img>`, `<script>`) reaching a *future* `{capture: true}` listener — except the actual listener is registered non-capturing, so that guard was already unreachable dead code for the current wiring. Writing the tests surfaced it as a *real* bug anyway: Vitest's jsdom environment can bind the bare `window` identifier inside one module differently from the `window` an event actually dispatches against, so the identity check silently discarded every single dispatched test event, no matter how the event was constructed. Removed the guard rather than chase the identity mismatch further — it wasn't protecting anything live. **General lesson: an identity comparison against a global like `window` is more fragile than it looks across module/bundler/test-environment boundaries; prefer `instanceof` or duck-typing checks if a target check is ever genuinely needed.** Separately, useful jsdom trivia for the next person writing a test that dispatches a synthetic `'error'` event on `window`: (1) a plain `Event` with `error`/`message` properties bolted on via `Object.assign` is silently ignored by every `addEventListener('error', ...)` listener — must be a genuine `new ErrorEvent(...)`; (2) an unhandled `'error'` event with no `preventDefault()` becomes a real uncaught exception in the *test process* in jsdom, unlike real browsers where the default action is just a console log — attach a throwaway listener that calls `preventDefault()` around each dispatch to suppress it.
