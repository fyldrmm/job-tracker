#### A first attempt broke and was fully reverted — here's what happened and what changed the second time

The first build wired `change-password` straight into the UI without testing it against the real deployed function first. It crashed in the app with a generic "Edge Function returned a non-2xx status code" error for both right and wrong passwords, and diagnosing it took a long back-and-forth (terminal quoting issues with `curl`, shell history-expansion breaking on a password containing `!`, wrong assumptions about which token to use) before finding the real root cause was still unconfirmed. Rather than keep patching blind, the whole M7 attempt was reverted with `git checkout ba09e92` (the last fully-verified commit) and rebuilt from scratch, this time with a hard rule: **every Edge Function action gets curl-tested standalone, with a real account, before any UI touches it.**

That discipline caught the actual bug fast the second time: `supabase.auth.updateUser()` and `supabase.auth.signOut()` both throw `"Auth session missing!"` when called on a client built from just an `Authorization` header override (`global.headers`) — they need an actual session set via `setSession()`, which requires a refresh token the Edge Function never has (only the access token, via the header). Confirmed by hitting the exact error live, not guessed. Fixed by:
- Using the **service-role admin API** (`admin.auth.admin.updateUserById`) for the password change itself, since the caller's identity was already independently verified via `getUser()` and the password check before this point.
- Using a **raw `fetch()` to `/auth/v1/logout?scope=others`** (the same REST endpoint the JS SDK calls internally) instead of `supabase.auth.signOut()`, since that only needs the bearer token in the header, not the SDK's internal session state.

Sources checked before implementing (not guessed): [Supabase docs — Signing out](https://supabase.com/docs/guides/auth/signout), [GitHub discussion on the "Auth session missing" error and `admin.updateUserById`](https://github.com/orgs/supabase/discussions/26791).

Both the `delete` and `change-password` cases were confirmed working via direct `curl` calls against the real deployed function (wrong-password rejection for both, and a real successful password change with `{"success":true}`) before `remoteStore.ts`/the UI were wired to it.

#### A second bug found during live QA itself: password-change silently broke deletion

After the above was deployed and curl-verified, the live QA pass (deleting the primary account and signing up fresh) hit a real, reproducible bug: after successfully changing the account's password through the app, trying to delete that same account failed with `{"error":"Not authenticated"}`.

Diagnosed live, not guessed: called Supabase's own `/auth/v1/user` endpoint directly with the exact token the browser was using, which returned `"error_code":"session_not_found","msg":"Session from session_id claim in JWT does not exist"` — the token wasn't expired, but its session had been revoked server-side. Checked GoTrue's (Supabase Auth's) own source code to confirm why: the `scope=others` logout handler falls back to a **full global logout** (every session, including the current one) if it can't resolve "which session made this request" from context — and that's exactly what the M7 change-password feature's raw `fetch()` to `/auth/v1/logout?scope=others` had been hitting. The very act of changing your password was silently logging you out entirely, not just other devices, blocking whatever you tried to do next in that same session.

Fixed by removing the sign-out-other-devices feature entirely, rather than continuing to debug a mechanism that had just caused confirmed real damage to a live account. It was scoped-up beyond the original ask during planning; password-change itself doesn't depend on it. **User-facing impact if this had shipped further:** every user who changed their password would have been unexpectedly signed out immediately after, with no indication why.

Confirmed working end-to-end after the fix: fresh sign-up with a compulsory name, name showing in the sidebar, name editing, password change with wrong/right current password, and account deletion — all live, all by the user, on real accounts.

#### Bug found and fixed during the QA round: structured-output schema rejected by Anthropic

First redeploy of the employment_type/work_mode extraction returned a generic "Extraction failed" (the function's deliberately-generic client-facing message for any non-2xx from Anthropic). Root cause: the schema expressed a nullable enum as `{ type: ['string', 'null'], enum: ['full_time', ..., null] }` — valid JSON Schema, but not a construct Anthropic's structured-output validator accepted, unlike the plain `type: ['string', 'null']` (no `enum`) used successfully by the original six fields. Fixed by switching to the documented-safe `anyOf` pattern: `{ anyOf: [{ type: 'string', enum: [...] }, { type: 'null' }] }`. Confirmed working after redeploy — worth remembering as a general pattern if a future field ever needs a nullable enum in a structured-output schema.

- **Multi-tracker note — naming:** the new per-tab entity is called a "tracker" in code (table `trackers`, type `Tracker`, hook `useTrackers`) specifically to avoid colliding with the existing `Board` component name, which is the whole page shell (sidebar + header + the currently-active tracker's 4 columns). - **Multi-tracker note — auto-create-default-tracker removed entirely:** `useTrackers` originally auto-created a "My Applications" tracker any time a user's list came back empty, guarded by a `useRef` lock (`creatingDefaultRef`) added after StrictMode's dev-mode double-invoke of effects caused two concurrent `refresh()` calls to both create one, producing duplicates. - **Audit note — D4/M6, the stale pending-signup flag (bug found, NOT fixed, user's explicit call):** tracing D4 (sign-up with an already-registered email) surfaced that `markPendingSignup()` (`src/hooks/useAuth.ts:39`) fires unconditionally on any non-error `signUp()` return — and Supabase's anti-enumeration design means an already-registered email *is* a non-error return, with no account created and no session ever arriving. - **Audit note — M5, session revocation on password change (shipped in `693ada3`, deployed and live-verified):** M7 documented a deliberate "NOT doing: signing out other sessions after a password change" after a `scope=others` logout fell back to a full global logout and broke a live account. Root cause, confirmed by reading `@supabase/auth-js`'s actual installed source (`GoTrueAdminApi.signOut`) rather than guessing: `admin.auth.admin.signOut(jwt, scope)` is a thin wrapper over `POST /auth/v1/logout?scope=X`, and `scope=others` needs GoTrue to resolve "which session is the calling one" from the JWT so it can exclude it — that resolution is exactly what failed live. `scope=global` has no such resolution step at all; it's documented as unconditionally terminating every session for the user, which is also exactly what M5 wants (kill everything including the caller's own session, then force a clean re-login) — so this isn't a workaround for the old bug, it sidesteps the entire failure class by not depending on the thing that broke. `account-action/index.ts`'s `change-password` case now calls `admin.auth.admin.signOut()` with the caller's own already-validated access token (stripped of the `Bearer ` prefix) and `scope: 'global'`, after the password update succeeds; a revoke failure is logged but doesn't fail the response, since the client forces its own local sign-out unconditionally regardless of what the server did. **Confirmed caveat, from Supabase's own docs, not assumed:** this revokes refresh tokens (blocks future refresh and any future login with the old password) but does **not** retroactively invalidate an already-issued access token before its own `exp` claim — "the user won't be immediately logged out and will only be logged out when the Access Token expires." So the UI copy ("Password changed — please log in again.") is accurate about *this* browser (which is force-signed-out client-side, instantly, independent of that caveat) but a stolen token elsewhere isn't necessarily dead the same second — it stops being renewable, not instantly killed. Worth knowing if this is ever explained to a user as "kicks out an attacker immediately" — the docs only promise "eventually, on next refresh," not instant. **Deployed and live-verified**, following the same discipline as the original M7 fix (curl-tested `change-password` standalone against a real account first — wrong password rejected, right password succeeds — before the UI path was trusted). **Empirical result, worth recording since it's better than the documented floor:** with a second real session open elsewhere, that session died *immediately* on password change, not merely on its next refresh — so in practice, on this Supabase project, `scope=global` behaves better than the docs' worst-case caveat above promises. Keep the caveat in the code comments and this note as the documented floor to design against (don't remove it or claim instant-kill is guaranteed), but it's worth knowing the observed behavior here was stronger than that floor.

- **Audit note — M2 atomic extraction quota (code shipped in `7fb4ca8`, dashboard steps still pending):** the old flow read both counts, compared in JS (`checkQuota` in the now-deleted `quota.ts`), called Anthropic, then inserted — a classic TOCTOU race, since concurrent requests all read the same pre-insert counts and all pass. Replaced with `reserve_extraction()`, a `SECURITY DEFINER` Postgres function (`0008_reserve_extraction.sql`) that checks both counts and inserts the event row in the same transaction, serialized by `pg_advisory_xact_lock(hashtext('reserve_extraction'))` — one global lock, not per-user, since extraction volume is far too low to need finer granularity. **The grant is the security-critical part, not boilerplate:** Postgres grants `execute` to `PUBLIC` by default on new functions, and `p_user_id` is a caller-supplied parameter — an ungranted-correctly version would let any authenticated browser client insert `extraction_events` rows for *any* user, or pass inflated limits to bypass the caps outright. The migration explicitly revokes from `public, anon, authenticated` and grants only to `service_role`; `index.ts` calls it via the admin client, never reachable from the browser. On any failure after the reservation (network error, non-2xx, `stop_reason: 'refusal'`, no content block, malformed JSON), `releaseReservation()` deletes the reserved row so a failed call doesn't burn quota it never spent — wired into all five failure exits, not just the obvious one. **Known accepted gap, not a bug:** if the Edge Function process dies between reserving and either succeeding or releasing (timeout, OOM, a mid-flight redeploy), the reserved row survives with null tokens and nothing ever compensates. Deliberately not building automatic cleanup for this — it leaks a few units of quota in a rare crash, which is a smaller problem than the race it replaces. If it's ever visibly annoying, the fix is a manual "delete extraction_events where input_tokens is null and created_at < now() - interval '1 hour'" in the SQL editor, not new code. `quota.ts`/`quota.test.ts` are deleted — the only Edge Function unit test in the repo, traded for live QA against the RPC (pre-agreed in the plan). **Not yet done:** the `0008` migration hasn't been run in the dashboard, `extract-job-details` hasn't been redeployed (and `quota.ts` needs deleting from its file list there too, or the stale copy keeps getting bundled), and none of it has been live-verified. - **C4 note — global error handlers, and a real bug the tests found:** see `src/lib/globalErrors.ts` (`d2db730`) for the design (module-level pub/sub, not React context — zero contexts exist in this codebase and there's exactly one consumer). Worth a general lesson: `handleError` originally had `if (event.target && event.target !== window) return`, meant to guard against non-bubbling resource-load errors (`<img>`, `<script>`) reaching a *future* `{capture: true}` listener — except the actual listener is registered non-capturing, so that guard was already unreachable dead code for the current wiring. Writing the tests surfaced it as a *real* bug anyway: Vitest's jsdom environment can bind the bare `window` identifier inside one module differently from the `window` an event actually dispatches against, so the identity check silently discarded every single dispatched test event, no matter how the event was constructed. Removed the guard rather than chase the identity mismatch further — it wasn't protecting anything live. **General lesson: an identity comparison against a global like `window` is more fragile than it looks across module/bundler/test-environment boundaries; prefer `instanceof` or duck-typing checks if a target check is ever genuinely needed.** Separately, useful jsdom trivia for the next person writing a test that dispatches a synthetic `'error'` event on `window`: (1) a plain `Event` with `error`/`message` properties bolted on via `Object.assign` is silently ignored by every `addEventListener('error', ...)` listener — must be a genuine `new ErrorEvent(...)`; (2) an unhandled `'error'` event with no `preventDefault()` becomes a real uncaught exception in the *test process* in jsdom, unlike real browsers where the default action is just a console log — attach a throwaway listener that calls `preventDefault()` around each dispatch to suppress it.
