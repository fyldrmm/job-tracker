6. **End each milestone in a runnable, committed state.** The app should launch and the milestone's slice should work before you stop.

1. **Sign-out didn't clear the local cache** — fixed in `Board.tsx`'s `handleSignOut` (clears local store, resets active tracker, then signs out). User confirmed fixed (re-ran the checklist, sign-out step passed).

3. **Data looked lost until a manual page reload after signup/login** — found during the SAME checklist re-run that confirmed #1 and #2. #### A first attempt broke and was fully reverted — here's what happened and what changed the second time

The first build wired `change-password` straight into the UI without testing it against the real deployed function first. It crashed in the app with a generic "Edge Function returned a non-2xx status code" error for both right and wrong passwords, and diagnosing it took a long back-and-forth (terminal quoting issues with `curl`, shell history-expansion breaking on a password containing `!`, wrong assumptions about which token to use) before finding the real root cause was still unconfirmed. Rather than keep patching blind, the whole M7 attempt was reverted with `git checkout ba09e92` (the last fully-verified commit) and rebuilt from scratch, this time with a hard rule: **every Edge Function action gets curl-tested standalone, with a real account, before any UI touches it.**

That discipline caught the actual bug fast the second time: `supabase.auth.updateUser()` and `supabase.auth.signOut()` both throw `"Auth session missing!"` when called on a client built from just an `Authorization` header override (`global.headers`) — they need an actual session set via `setSession()`, which requires a refresh token the Edge Function never has (only the access token, via the header). Confirmed by hitting the exact error live, not guessed. Fixed by:
- Using the **service-role admin API** (`admin.auth.admin.updateUserById`) for the password change itself, since the caller's identity was already independently verified via `getUser()` and the password check before this point.
- Using a **raw `fetch()` to `/auth/v1/logout?scope=others`** (the same REST endpoint the JS SDK calls internally) instead of `supabase.auth.signOut()`, since that only needs the bearer token in the header, not the SDK's internal session state.

Sources checked before implementing (not guessed): [Supabase docs — Signing out](https://supabase.com/docs/guides/auth/signout), [GitHub discussion on the "Auth session missing" error and `admin.updateUserById`](https://github.com/orgs/supabase/discussions/26791).

Both the `delete` and `change-password` cases were confirmed working via direct `curl` calls against the real deployed function (wrong-password rejection for both, and a real successful password change with `{"success":true}`) before `remoteStore.ts`/the UI were wired to it.

#### A second bug found during live QA itself: password-change silently broke deletion

After the above was deployed and curl-verified, the live QA pass (deleting the primary account and signing up fresh) hit a real, reproducible bug: after successfully changing the account's password through the app, trying to delete that same account failed with `{"error":"Not authenticated"}`.

Diagnosed live, not guessed: called Supabase's own `/auth/v1/user` endpoint directly with the exact token the browser was using, which returned `"error_code":"session_not_found","msg":"Session from session_id claim in JWT does not exist"` — the token wasn't expired, but its session had been revoked server-side. Checked GoTrue's (Supabase Auth's) own source code to confirm why: the `scope=others` logout handler falls back to a **full global logout** (every session, including the current one) if it can't resolve "which session made this request" from context — and that's exactly what the M7 change-password feature's raw `fetch()` to `/auth/v1/logout?scope=others` had been hitting. The very act of changing your password was silently logging you out entirely, not just other devices, blocking whatever you tried to do next in that same session.

Fixed by removing the sign-out-other-devices feature entirely, rather than continuing to debug a mechanism that had just caused confirmed real damage to a live account. It was scoped-up beyond the original ask during planning; password-change itself doesn't depend on it. **User-facing impact if this had shipped further:** every user who changed their password would have been unexpectedly signed out immediately after, with no indication why.

Confirmed working end-to-end after the fix: fresh sign-up with a compulsory name, name showing in the sidebar, name editing, password change with wrong/right current password, and account deletion — all live, all by the user, on real accounts.

The audit flagged the consequence (M5): changing your password because you suspect compromise does not actually evict an attacker. **The approved plan reverses that decision** — revoke *all* sessions including the caller's, then force a clean re-login, which sidesteps the "which session is current" scoping that failed before. Not yet implemented. When it is, verify the exact admin revoke call against supabase-js/GoTrue and curl-test it standalone *before* wiring any UI — that discipline is the whole reason the second M7 attempt succeeded.
- **Incident — `applications` table emptied, cause never established:** during this session every row in `applications` disappeared (both trackers survived, so it was not a tracker-delete cascade). Investigated thoroughly and **not solved**: the app's own code issues no unscoped delete (every path in `remoteStore.ts` is `.eq('id', id)`), the delete-tracker confirm correctly required typing the tracker name and the user did not type it, and Postgres/API logs for the window contained no `DELETE`/`TRUNCATE` against `applications`. The logs *did* reveal an unrelated, previously-undocumented event: something ran a first-time Supabase CLI migration push (created `supabase_migrations.schema_migrations`, then failed on `create type application_stage` → `already exists`) — but none of the six migration files contain a destructive statement, so that cannot explain it either. Data was test-only and the user chose to stop investigating. **If it recurs with real data, check API logs for a `DELETE /rest/v1/applications` without an `id=eq.` filter — that's the shape that matches the evidence (RLS would scope such a delete to just that user's rows).**
- **Note — rotating the Supabase secret key invalidates live sessions:** after the user generated a new `sb_secret_…` key and deleted the old default, the signed-in app started returning "Not authenticated" from `account-action`, and the board rendered empty because `useAuth()` saw no user and fell back to guest mode (reading the empty local cache, not the account's data). Signing out and back in fixed it completely. Nothing was wrong with the data. Expect a forced re-login after any key rotation, and recognize "my data is gone" reports of this shape as a session problem, not a data problem.
